# 파일럿 리뷰: 나루토 1~5화

전체 파이프라인(자막 파싱 → LLM 추출 → 온톨로지 병합 → Neo4j 내보내기)을 100화 전체로
확장하기 전에, 나루토 1~5화로 먼저 검증한 결과를 정리한다.

## 업데이트: 품질 이슈 수정 후 재검증 (2차)

아래 "발견된 품질 이슈" 1, 2번을 고치고 나서 `--force`로 1~5화를 재추출해 다시
확인했다. 3번(사건 설명)은 프롬프트만 강화했다.

- **(이슈 2 수정) 타입 충돌 시 기존 엔티티 재사용**: `ontology/builder.py`가 경고만
  남기던 것에서, 같은 이름이 이미 다른 클래스로 존재하면 **잘못된 타입의 중복
  엔티티를 만들지 않고 기존 엔티티를 재사용**하도록 바꿨다. 재추출 결과 "나루토",
  "구미호"가 더 이상 `Location`으로 잘못 생성되지 않고, 실제로 `닌자 일족`/`호카게`/
  `마을`/`우치하 일족`/`제 7반` 등 6건의 충돌이 모두 기존 엔티티로 정상 재사용됐다
  (stderr 경고로 계속 확인 가능).
- **(이슈 1 수정) 존칭 접미사 정규화로 동일 인물 병합 개선**: "~선생님/~님/~쨩/~군/
  ~양/~씨"를 매칭 시에만 떼어내는 정규화(`_match_key`)를 추가하고, 추출 프롬프트에도
  "성+이름을 대표 이름으로, 존칭이 붙은 형태는 별칭으로 넣어라"는 지침을 추가했다.
  재추출 결과 캐릭터 수가 19 -> 15로 줄었고, 특히 "카카시 선생님"과 "하타케 카카시"가
  이번에는 처음부터 하나의 엔티티(`카카시`, 별칭 `선생님`/`카카시 선생님`)로
  추출되어 별도 수정 로직 없이도 병합됐다. "호카게님"도 별칭 "3대째 호카게님"을
  포함해 하나로 묶였다.
- **재실행 안전성 재확인**: 수정 후에도 캐시가 있으면 동일한 `naruto.ttl`(md5 동일)이
  나오는 idempotency를 다시 확인했다.

| 클래스 | 1차 | 2차(수정 후) |
|---|---|---|
| Character | 19 | 15 |
| Location | 10 | 11 |
| Organization | 5 | 5 |
| Clan | 0 | 2 |
| Skill | 14 | 17 |
| InnateAbility | 5 | 4 |
| Item | 8 | 7 |
| Artifact | 5 | 5 |

| 전체 트리플 | 431 | 440 |

(트리플 총량은 비슷하지만 Character가 줄어든 것은 중복 제거 효과, 나머지 항목의
소폭 변화는 로컬 LLM 출력이 매번 완전히 동일하지는 않은 데서 오는 자연스러운 편차다.)

### 남아있는 한계 (이번에 고치지 않음)

- **처음부터 잘못 분류되는 경우는 여전히 못 잡음**: 타입 충돌 재사용은 "같은 이름이
  이미 다른 클래스로 존재할 때"만 동작한다. 이번 재추출에서 "코노하마루"(실제로는
  3대 호카게의 손자인 캐릭터)가 `Location`으로, "손자님 (3대째 호카게님의 손자)"처럼
  설명문이 이름 자리에 들어간 캐릭터가 새로 생기는 등, **처음 등장할 때부터 잘못
  분류/서술되는 경우**는 구조적으로 감지할 방법이 없다. 이건 규칙 기반으로 막기
  어렵고, 더 나은 모델을 쓰거나 사람이 주기적으로 검토하는 수밖에 없다.
- **완전히 다른 표기의 동의어는 여전히 안 묶임**: "제 7반"과 "7반", "닌자 아카데미"와
  "나뭇잎 마을 닌자 학원"처럼 존칭이 아니라 표기 자체가 다른 동의어는 이번 정규화로
  해결되지 않는다(의도한 범위 밖).
- 사건(Event) 설명은 프롬프트 지침 추가 후 이전보다 구체적인 사건 위주로 나오는
  경향이 보이지만("하위닌자 선발 테스트", "미즈키가 나루토를 속여 두루마리를 노리는
  사건" 등), "후퇴하지 않고 초난해한 시련을 클리어하기 위해 노력함"처럼 여전히
  서술적인 항목이 일부 섞여 있다.

결론: 결정론적으로 고칠 수 있는 유형(존칭 차이, 타입 오분류로 인한 중복 생성)은
코드로 개선했고, 근본적으로 모델의 텍스트 이해 오류에 기인하는 유형(첫 등장부터
잘못 분류, 완전히 다른 표기의 동의어)은 여전히 남아 있다. 100화 전체 확장 여부는
이 잔여 한계를 감수할지, 더 큰 모델로 바꿔볼지 사용자가 판단할 부분이다.

## 최초 파일럿 (1차)

## 실행 정보

- 명령: `python scripts/run_episode.py naruto --episodes 1-5`
- LLM: `.env`의 기본값인 로컬 Ollama(`gemma4:latest`), API 키 없이 실행
- 소요 시간: 약 6분 14초 (5화, 화당 2개 청크 = 총 10회 LLM 호출)
- 결과 파일: `data/naruto/ontology/naruto.ttl` (인스턴스 트리플 431개)

## 정량적 결과

| 클래스 | 개수 |
|---|---|
| Character | 19 |
| Location | 10 |
| Organization | 5 |
| Skill | 14 |
| InnateAbility | 5 |
| Item | 8 |
| Artifact | 5 |
| Event | 14 |

Neo4j로 내보낸 결과도 동일하게 노드 85개, 관계 179개로 확인했고(`scripts/export_neo4j.py`),
재실행해도 노드/관계 수가 그대로 유지되어(MERGE 기반) idempotent함을 확인했다.

## 잘 동작한 부분

- **화를 넘나드는 동일 인물 병합**: 이름 색인이 여러 화에 걸쳐 잘 동작해 "우즈마키
  나루토"(별칭 "나루토"), "하루노 사쿠라"(별칭 "사쿠라쨩"/"사쿠라"), "우치하 사스케"
  (별칭 "사스케군"/"사스케")가 1~5화 전체에서 하나의 엔티티로 정확히 누적됐다.
  `mentionedIn`으로 각 인물이 몇 화에 등장했는지도 그대로 추적된다.
- **파이프라인 재실행 안전성**: 캐시가 있으면 LLM을 다시 부르지 않고, 온톨로지 파일도
  같은 입력이면 바이트 단위로 동일하게 재생성됨을 확인했다(idempotent).
- **로컬 LLM만으로도 최소 동작**: API 키 없이 Ollama만으로 캐릭터/장소/기술/사건/관계가
  실제로 추출되어, Provider 추상화가 의도대로 "누구나 바로 시작할 수 있는" 기본값
  역할을 한다.

## 발견된 품질 이슈

### 1. 다른 호칭으로 불리는 동일 인물이 별도 엔티티로 남음

- 예: "호카게님"(1화) / "4대 호카게"(1화) / "3대째 호카게님"(2화)가 별도 엔티티로 존재.
  (이 중 "4대 호카게"는 실제로 다른 인물이지만, "호카게님"과 "3대째 호카게님"은
  같은 인물일 가능성이 높다.)
- 예: "카카시 선생님"(3화)과 "하타케 카카시"(별칭 "카카시", 4~5화)가 병합되지 않음.
- 원인: 병합 로직이 완전 일치(대소문자 무시) 이름/별칭만 같은 엔티티로 인식한다.
  칭호("~님", "~선생님", "N대째")가 붙거나 성/이름 표기가 갈리면 다른 엔티티로 취급된다.

### 2. LLM이 관계의 predicate와 대상 타입을 잘못 짝짓는 경우

- 예: "나루토"(캐릭터)가 `current_location`의 대상으로 쓰여 `Location` 타입으로도
  생성될 뻔했고, "구미호"(캐릭터)가 `Location`/`Artifact` 대상으로도 등장했다.
- 이번 리뷰에서 `ontology/builder.py`에 **타입 충돌 경고**를 추가해, 같은 이름이 이미
  다른 클래스로 존재하는데 새 클래스로 또 생성 요청되면 stderr에 경고를 남기도록 했다
  (동작 자체를 막지는 않음 - 무엇이 애매한지 드러내는 목적).
  ```
  경고: '나루토'이(가) 이미 Character 타입으로 존재하는데 Location 타입으로도
  생성 요청되었습니다. LLM이 관계의 predicate/대상을 잘못 짝지었을 수 있습니다.
  ```
- 1~5화 재처리에서 이런 경고가 7건 발생했다(나루토/구미호/인술/그림자 분신술/방울/제 7반).

### 3. 사건(Event) 설명이 다소 서술적/거칠음

- "나루토가 양친 사별과 인정받지 못함으로 겪은 고통." 처럼 사건이라기보다 서술 요약에
  가까운 항목이 섞여 있다. 사건 단위 추출 기준을 프롬프트에서 더 좁혀야 할 수 있다.

## 다음 단계 제안 (이번 파일럿 범위 밖)

- **더 큰/특화 모델과 비교**: 같은 1~5화를 OpenAI 등 더 큰 모델로 돌려 로컬 모델과
  추출 품질을 비교해볼 가치가 있다 (Provider Proxy 구조 덕분에 `.env`의
  `LLM_PROVIDER` 값만 바꾸면 됨).
- **호칭 정규화**: "~님", "~선생님", "N대째" 같은 패턴을 프롬프트에서 대표 이름으로
  통일하도록 지시하거나, 별도 후처리 규칙을 추가하는 방향을 검토.
- **타입 충돌 항목의 사람 검토 큐**: 현재는 경고만 출력하는데, 나중에는 충돌 목록을
  별도 파일로 쌓아 사람이 주기적으로 검토/병합할 수 있게 하면 좋을 것 같다.

## 결론

100화 전체로 확장하기 전에 목표했던 "전체 흐름이 끊기지 않고 idempotent하게 동작하는지"는
확인됐다. 위 품질 이슈들은 파이프라인 구조를 바꿔야 하는 문제라기보다, 프롬프트/모델
선택/후처리 규칙으로 개선할 수 있는 성격이라 판단해 이번 파일럿에서는 구조를 바꾸지 않고
기록만 남긴다.
